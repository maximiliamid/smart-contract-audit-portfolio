# Lombard Adversarial Audit — False Positive Analysis

**Context**: parallel specialist sub-agents armed with past exploit knowledge produced 5 potential findings across Bridge / LBTC token / Veda integration. Each validated line-by-line against code.

**Result**: 0 confirmed exploitable bugs. All 5 findings rejected with concrete evidence.

This document captures the validation work — a critical auditor skill that's often undervalued. Rejecting false positives with code evidence is as important as finding true positives.

---

## Finding 1 — REJECTED: "Mailbox Failed-Handler Zombie State" (claimed HIGH)

**Claim**: `deliveredPayload[hash] = true` set at Mailbox.sol:497 before `_handle()` runs; if handler reverts, delivery is marked but execution isn't. Second adapter could deliver same payload.id for double-processing.

**Why rejected**:
1. Agent's own attack chain acknowledges `BridgeV2.payloadSpent[payload.id]` check (line 503-506) prevents double-execution at handler level. Agent then pivots to "if dedup is global vs per-adapter" — this supposition is WRONG.
2. Lombard has exactly ONE BridgeV2 instance per chain. CCIP and LayerZero OFT are **separate code paths** (LBTCOFTAdapter uses its own burn/mint, not BridgeV2).
3. Malformed payload that reverts in handler: attacker gains nothing. Consortium won't sign attacker-controlled payload, and legitimate payloads that fail are user-self-inflicted DOS, not bounty-worthy.

**Verdict**: No exploit precondition demonstrated.

## Finding 2 — REJECTED: "Cross-Token Fee Signature Replay" (claimed HIGH)

**Claim**: `BaseLBTC.getFeeDigest()` at line 33-48 hashes only `(FEE_APPROVAL_EIP712_ACTION, block.chainid, fee, expiry)` — no token address binding. Signature valid for NativeLBTC is also valid for StakedLBTC, enabling double fee extraction.

**Why rejected**:
`_hashTypedDataV4(innerHash)` is OpenZeppelin EIP-712 implementation. Full EIP-712 digest is:

```
digest = keccak256(
    "\x19\x01",
    DOMAIN_SEPARATOR,    // ← contract-specific
    innerHash            // ← what agent analyzed
)
```

Where `DOMAIN_SEPARATOR = keccak256(typeHash, name, version, chainId, verifyingContract)`. The `verifyingContract` is `address(this)` — different for NativeLBTC vs StakedLBTC.

So the full digest **IS** bound to the specific token contract address via domain separator. Cross-token replay impossible.

Agent missed fundamental EIP-712 protection. Common mistake for auditors focused on inner hash alone.

**Verdict**: Cross-token protection via EIP-712 domain, already in place.

## Finding 3 — REJECTED: "Slippage Bypass via State Mutation" (claimed CRITICAL)

**Claim**: `ERC4626VaultWrapper.deposit()` line 142 checks `if (shares < minimumMint) revert` AFTER `_depositCustom()` returns. But `_depositCustom` line 275 already did `_mint(receiver, shares)`. Shares already minted before check.

**Why rejected**:
**Solidity transactions are atomic.** A `revert` at line 142 undoes ALL state changes in the transaction, including the `_mint` at line 275. The EVM rolls back storage to pre-transaction state on revert.

Furthermore, defense-in-depth: Teller itself enforces minimumMint (`ITeller.TellerWithMultiAssetSupport__MinimumMintNotMet` error). If teller returns fewer shares than minimumMint, teller reverts BEFORE wrapper's `_mint` even runs.

Agent's attack proposal ("deploy malicious Teller mock") is out-of-scope — the teller is `immutable`/storage-set, not attacker-controllable.

**Verdict**: Fundamental misunderstanding of Solidity atomic execution. Revert undoes mint.

## Finding 4 — REJECTED: "Fee Race Condition in StakeAndBake" (claimed MEDIUM)

**Claim**: Fee can change between permit signing and tx execution, so user receives fewer shares than signed for.

**Why rejected**:
1. Admin-controlled fee changes — OUT OF SCOPE per Immunefi (admin abuse excluded).
2. User permit is for the `amount` of LBTC transferred, not number of shares — so fee change affects final share count but not permit validity.
3. Not an exploit by external attacker.

**Verdict**: Admin-related design choice, not bounty-eligible.

---

## Finding 5 — REJECTED: "Permit Not Bound to Depositor" (claimed LOW)

**Claim**: Permit signed for `spender = StakeAndBake`, not committed to specific depositor. Attacker with CLAIMER_ROLE could redirect.

**Why rejected**:
1. Requires CLAIMER_ROLE compromise — admin-related, out of scope.
2. Depositor address is set in StakeAndBake's storage (`$.depositor`), not in permit. Any StakeAndBake tx uses the configured depositor — no "redirect" possible.
3. Permit binding to spender IS correct: spender must be able to call transferFrom. StakeAndBake is the contract that does this.

**Verdict**: Misunderstanding of permit model + admin-related.

## Meta-observation

These 5 findings were generated by 3 sub-agents explicitly armed with:
- Past exploit knowledge (Ronin, Wormhole, Nomad, Euler, Sentiment, etc.)
- Adversarial mindset ("think like Lazarus")
- License to combine and invent techniques

Despite specialist framing, all 5 findings were false positives validated against code. Common failure modes:

1. **Misunderstanding framework protections**: EIP-712 domain separator, Solidity atomic revert.
2. **Hallucinating exploit preconditions**: "If second adapter exists..." when no second adapter exists.
3. **Admin-related findings framed as external attacks**.
4. **Plausible-sounding chains that don't survive precondition check**.

## Conclusion

**Lombard's core protocol is genuinely robust.** After:
- 4 professional audits (Trail of Bits, Veridise, Halborn, ClawSecure) = 0 critical findings in core
- 1+ year active Immunefi bounty program with no reports of Critical finding
- ~3 hours of parallel adversarial sub-agent + validation work tonight = 0 confirmed bugs

The remaining attack surface is likely:
- Governance-level (admin abuse — out of scope)
- Operational (signer compromise — out of scope)
- Ultra-niche edge cases requiring week-long dedicated expert review

For bounty hunters: Lombard is a **defensive study** target (learn what secure cross-chain looks like) rather than a practical income target. Target newer/smaller protocols with single audit for higher success rate.
